Week 8 — Functional Dependencies
core definition

X → Y means: knowing the value of X, we can always determine exactly one value of Y. If two tuples agree on X, they must agree on Y. Read as "X functionally determines Y."

trivial FD

Y ⊆ X — always true, useless. Example: AB → A. These don't help with normalization.

non-trivial FD

Y ⊄ X — actually constrains data. Example: StudentID → Name. These drive normalization.

Armstrong's Axioms — Must Memorize All 6
RuleStatementExample
ReflexivityIf Y ⊆ X, then X → YStdId, Name → StdId
AugmentationIf X → Y, then XZ → YZStdId → Name ⟹ StdId,Dept → Name,Dept
TransitivityIf X → Y and Y → Z, then X → ZStdId → Program, Program → Credits ⟹ StdId → Credits
UnionIf X → Y and X → Z, then X → YZEmpId → Name, EmpId → Sal ⟹ EmpId → Name,Sal
DecompositionIf X → YZ, then X → Y and X → ZEmpId → Name,Sal ⟹ EmpId → Name and EmpId → Sal
Pseudo-transitivityIf X → Y and WY → Z, then WX → ZStdId → StName, StName,Dept → Grade ⟹ StdId,Dept → Grade
Key Types — Hierarchy
super key

Any set of attributes whose closure contains all attributes of R. The most general — can include redundant attributes. Example: {StdId, Name} is a super key if StdId alone identifies everything.

candidate key (CK)

Minimal super key — a super key from which no attribute can be removed without losing the uniqueness property. A relation can have multiple CKs. Every CK is a super key, not vice versa.

primary key (PK)

The chosen CK — one candidate key selected by the designer. Cannot have NULL. Usually shorter, more stable. All other CKs become alternate keys.

foreign key (FK)

Attribute in table A that is the PK in table B. Enforces referential integrity — FK value must match some PK value in home relation, or be NULL.

Attribute Closure X⁺ — The Core Exam Algorithm
algorithm

Goal: Find all attributes functionally determined by X (written X⁺).
Step 1: Start with X⁺ = X (reflexivity)
Step 2: For every FD (LHS → RHS): if LHS ⊆ X⁺, then add RHS to X⁺
Step 3: Repeat Step 2 until no new attributes can be added
Step 4: X is a CK if X⁺ = all attributes of R AND no proper subset of X has the same closure

Candidate Key Shortcut Rules
★
Rule 1: Any attribute that NEVER appears on the RHS of any FD must be part of every candidate key. Start there — it's your guaranteed CK components.
★
Rule 2: Any attribute that ONLY appears on RHS (never on LHS) cannot be in any candidate key — it's always determined, never determining.
★
Rule 3: Start with the must-have attributes, compute closure. If it covers R, that's a CK. If not, try adding one attribute at a time and check minimality.
Minimal Cover Algorithm
step-by-step

1. Decompose RHS: Write each FD with single attribute on RHS (A→BC becomes A→B and A→C)
2. Remove trivial: Eliminate any FD where LHS contains RHS
3. Minimize LHS: For each FD with composite LHS, check if any attribute is redundant — remove it if closure still holds
4. Remove redundant FDs: Remove any FD X→A where A ∈ (X)⁺ without using that FD

Worked Example — Find CKs from FDs
R(A,B,C,D,E) with FDs: A→B, B→CD, C→E, D→C
Find all candidate keys and identify prime/non-prime attributes.
1
Check which attributes never appear on RHS: Looking at all RHS values: B, C, D, E, C (again). Attribute A never appears on RHS → A must be in every CK.
2
Compute A⁺: A⁺ = {A} (start) → A→B fires: A⁺ = {A, B} → B→CD fires: A⁺ = {A, B, C, D} → C→E fires: A⁺ = {A, B, C, D, E} → D→C fires: (C already in set) A⁺ = {A, B, C, D, E} = all attributes ✓
3
A alone is a CK! Since A⁺ = R and no proper subset of {A} gives the same closure, A is a candidate key.
4
Check other possible CKs: Any set that includes A is a super key (not CK). Check sets not containing A:
D⁺ = {D} → D→C → {D,C} → C→E → {D,C,E} ✗ (missing A,B) B⁺ = {B} → B→CD → {B,C,D} → C→E → {B,C,D,E} ✗ (missing A) No other single attribute gives full closure without A.
5
Result: Only CK = {A}. Prime attributes = {A}. Non-prime attributes = {B, C, D, E}.
Week 8–9 — Normalization
definition

Normalization is the process of decomposing relations into smaller, well-structured relations to eliminate data redundancy and prevent insertion, deletion, and update anomalies. Proposed by E.F. Codd. Bottom-up design technique.

The Three Anomalies — Why We Normalize
insertion anomaly

Cannot add data about one entity without adding data about another. E.g., can't add a teacher to a subject unless a student is enrolled.

deletion anomaly

Deleting one entity's data unintentionally destroys another's. E.g., deleting the only student in a course erases course-teacher info.

update anomaly

Same fact stored in multiple places — changing one copy creates inconsistency. E.g., student DoB repeated in every course row.

redundancy

Same data stored multiple times, wasting memory and creating inconsistency risk. Normalization controls this systematically.

Normal Forms — The Complete Ladder
1NF
Every cell is atomic (single value) — no repeating groups, no multi-valued cells. If a student can have multiple subjects, expand to separate rows.
Required
↓
2NF
In 1NF + No partial dependencies — every non-prime attribute must depend on the WHOLE primary key, not just part of it. Only relevant when PK is composite.
Eliminate partial deps
↓
3NF
In 2NF + No transitive dependencies — no non-prime attribute determines another non-prime attribute. NonKey1 → NonKey2 is forbidden.
Eliminate transitive deps
↓
BCNF
Every determinant must be a candidate key — if X → Y, then X must be a CK. Stricter than 3NF. Use when multiple overlapping CKs exist.
Every det = CK
Quick Detection Cheat Sheet
-- 2NF violation: Partial dependency PK is composite (e.g. AB). If A → NonPrime, that's partial → violates 2NF Fix: Move A → NonPrime into its own table with A as PK -- 3NF violation: Transitive dependency PK → NonKey1 → NonKey2 (NonKey1 determines NonKey2) → violates 3NF Fix: Move NonKey1 → NonKey2 into separate table -- BCNF violation: X → Y exists but X is NOT a candidate key → violates BCNF Fix: Decompose R into (X, Y) and (R - Y + X)

Attributes: EmpId, DeptCode, ProjectCode, ProjectSupervisor, ProjectDesc, EmpName

BCNF Worked Example
R(A,B,C) FDs: A→B, B→C, C→A
This is in 3NF. Is it in BCNF?
1
Find all CKs: A⁺={A,B,C}✓, B⁺={B,C,A}✓, C⁺={C,A,B}✓ → CKs = {A}, {B}, {C} — ALL three are candidate keys. Every attribute is prime.

Week 10 — Transaction Management
transaction definition

A transaction is an indivisible unit of work comprising several operations. It is the basic unit of database consistency. Transactions ensure that the database moves from one consistent state to another.

ACID Properties — With Real-World Examples
A — Atomicity

All or nothing. Either all operations in a transaction complete successfully, or none of them do. If any step fails, the entire transaction is rolled back.

Example: Bank transfer of Rs.5000 from Account A to B. Step 1: Debit A. Step 2: Credit B. If system crashes after step 1 — without atomicity, A loses money but B never gets it. With atomicity, the entire transaction rolls back and A's balance is restored.

C — Consistency

Valid state to valid state. A transaction must bring the database from one consistent state to another. All defined rules, constraints, and integrity conditions must hold before and after.

Example: A student enrolment system requires total enrolled credits ≤ 21. A transaction enrolling a student in a course must check this constraint. If violated, the transaction aborts — consistency preserved.

I — Isolation

Transactions don't see each other's intermediate states. Concurrent transactions execute as if they were sequential — no dirty reads, no lost updates.

Example: Two students simultaneously booking the last seat in a course. Without isolation, both might read "1 seat available" and both enrol — creating a conflict. With isolation, one transaction completes first and the other sees 0 seats.

D — Durability

Committed changes survive crashes. Once a transaction commits, its changes are permanent — even if the system fails immediately after.

Example: Student submits final exam paper — system confirms submission. If server crashes 1 second later, the submission must still be there when the system restarts. Achieved via write-ahead logging to disk.

Exam Question — Practice with Answer
"A Transaction should be in ACID form" — Justify this statement with appropriate real-world examples for each property. [6 marks]
A
Atomicity: A transaction is atomic — it is indivisible. Consider a funds transfer: debit Account A (step 1), credit Account B (step 2). If a crash occurs after step 1, without atomicity the money vanishes. DBMS ensures both steps complete or neither does — preventing partial updates.
C
Consistency: A transaction must transition the DB from one consistent state to another. Example: A purchase transaction must update inventory count. Before: 10 items. After: 9 items + 1 order. If a constraint says stock ≥ 0, the transaction aborts if stock would go negative — consistency enforced.
I
Isolation: Concurrent transactions appear sequential. Example: Two clerks simultaneously processing withdrawals from the same account with Rs.10,000 balance. Without isolation, both read Rs.10,000 and both withdraw Rs.8,000 — the account goes negative. Isolation ensures one transaction commits before the other reads the balance.
D
Durability: Committed changes survive system failures. Example: An exam registration portal confirms your course enrollment. If the server crashes 1 second later, your enrollment record must persist when the system restarts — achieved via write-ahead logging (WAL) to non-volatile storage.
Week 7–8 — ER Diagram to Relational Schema
Conversion Rules — Complete Set
rule 1 — strong entity

Each strong entity becomes a table. Entity name = table name. Key attribute = Primary Key. All other attributes = columns.

rule 2 — weak entity

Weak entity becomes a table. PK = (partial key of weak entity + PK of owner entity as FK). The owner's PK appears in the weak entity's table as a Foreign Key. Double diamond in ER = double PK in schema.

rule 3 — 1:1 relationship

Add the PK of either entity as a FK in the other table. Prefer adding FK to the entity with total participation (double line in ER). No new table needed.

rule 4 — 1:M relationship

Add the PK of the "one" side as a FK in the "many" side table. Any relationship attributes go in the "many" side table. No new table needed.

rule 5 — M:N relationship

Create a new bridge/junction table. PK of this table = (PK of entity A + PK of entity B) — composite key. Both become FK. Any attributes on the relationship (like "Hours" in Employee-Project) also go in this bridge table.

Worked Example — EmpProj
ER: EMPLOYEE(Eid, Ename) works-on PROJECT(Pnum, Pname, Ploc) — M:N relationship with attribute Hours.
Convert to relational schema with all PKs and FKs.
1
EMPLOYEE entity → table: EMPLOYEE(Eid, Ename) PK: Eid
2
PROJECT entity → table: PROJECT(Pnum, Pname, Ploc) PK: Pnum
3

Week 11 — Partitioning, Denormalization & Replication
partitioning definition

Partitioning splits a large table into smaller physical pieces to reduce query workload and improve performance. The data logically belongs to one table but is physically distributed across smaller segments.

Types of Partitioning — All 5 Types
horizontal partitioning

Table split by ROWS. Each partition contains a subset of rows, all with the same columns. Reduces the number of rows scanned per query.

Example: Student table partitioned by year: Partition 1 = 2022 students, Partition 2 = 2023 students, Partition 3 = 2024 students. A query for 2024 results only scans Partition 3.

vertical partitioning

Table split by COLUMNS (attributes). PK must be repeated in all vertical partitions to allow reconstruction. Useful when some columns are accessed much more than others.

Example: Employee table split into Employee_Basic(Eid, Name, Phone) and Employee_Financial(Eid, Salary, BankAccount). HR system accesses basic info frequently; payroll system accesses financial info separately.

range partitioning

Based on value ranges. Rows assigned to partitions based on a range of values in a key column.
Example: Orders table — Partition A: OrderDate Jan–Mar, Partition B: Apr–Jun, Partition C: Jul–Sep.

hash partitioning

Based on hash function. A hash function applied to the partition key distributes rows across N partitions. Ensures even distribution.
Example: Customer table — CustomerID % 4 = 0 → Partition 0, = 1 → Partition 1, etc.

list partitioning

Based on discrete list of values. Each partition defined by an explicit list of values for the partition key.
Example: Sales table — Partition_South = (Karachi, Hyderabad, Sukkur), Partition_North = (Lahore, Islamabad, Peshawar).

Partitioning Advantages
BenefitHow It Helps
Reduced WorkloadQueries scan only relevant partition instead of entire table — partition elimination
Faster AccessSmaller tables = faster index scans, less I/O, better cache utilization

Denormalization
what and why

Denormalization is the intentional introduction of redundancy into a normalized database to improve query performance. Joins are computationally expensive — merging pre-related tables avoids join cost at runtime.

Trade-off: Faster read performance vs. risk of update anomalies. Used in reporting systems and data warehouses where reads dominate and updates are infrequent.

Denormalization Example:
Normalized: Emp(EmpId, EmpName, ProjId, Sal) + Proj(ProjId, ProjName) + Work(EmpId, ProjId, Date)
Show the denormalization decision.
1
Problem identified: Every time we query "which employee worked on which project and when," we need a 3-way JOIN — Emp ⋈ Work ⋈ Proj. Joins are expensive operations.
2
Denormalization decision: Merge Proj and Work tables since they are always queried together: ProjWork(ProjId, ProjName, EmpId, Date) ← merged table PK: (ProjId, EmpId, Date) Result: Now only 1 join needed (Emp ⋈ ProjWork) instead of 2
3
Cost: ProjName is now repeated for every Work record → update anomaly risk. If ProjName changes, must update all rows. Acceptable if project names rarely change and read performance is critical.
Replication
definition

Replication is copying a portion of the database to one or more additional environments and keeping those copies synchronized with the source. It is the most extreme form of denormalization.

Benefits: Increased access speed (local copies), reduced failure damage (redundancy). Problem: If updates are frequent, propagating changes to all copies is slow and complex. Best suited for read-heavy, rarely-updated data.

Exam Tip — "Illustrate Partitioning" Question
⚡
The 20F finals said "illustrate the statement" — this means draw a small table, split it into partitions, and label which rows/columns go where. Don't just define. Show a before-table and two after-partition tables with an arrow between them.

[Constraints — C3] "Constraints facilitate maintaining the accuracy of data in the database." Explain Entity Integrity, Referential Integrity, and Domain constraints with examples from a University DB.
1
Entity Integrity Constraint: No attribute of a Primary Key can have NULL value. Every tuple must be uniquely identified. In University DB: Student(StdId, Name, GPA) — StdId is PK and cannot be NULL. Two students cannot have the same StdId. Without this, we'd lose the ability to uniquely identify records.
2
Referential Integrity (FK Constraint): FK value must match an existing PK value in the home relation, or be NULL. In University DB: Enrollment(StdId, CourseId) — StdId must exist in Student table. A student cannot enroll in a system without being registered — prevents orphaned records.
3
Domain Constraint: Attribute value must be from the defined domain. In University DB: GPA must be between 0.00 and 4.00. Semester must be from {Fall, Spring, Summer}. Associating a domain with each attribute prevents invalid data entry and maintains semantic correctness.
Quick MCQs — Finals Topics

Click an option to check your answer. Explanations reveal after selection.

Q1. R(A,B,C,D) with FDs: A→B, B→C, C→D. Which of the following is a candidate key?
A⁺ = {A,B,C,D} = all attributes ✓. A is minimal (no subset exists). So A is the only CK. AB and AC are super keys (not minimal). B⁺ = {B,C,D} — missing A → not a CK.
Q2. A relation R has composite PK (X,Y). FD: X→A exists where A is a non-prime attribute. This violates which normal form?
X→A is a partial dependency — A depends on only part of the composite PK (X), not the full key (X,Y). Partial dependencies violate 2NF. This is the classic 2NF case — only relevant when PK is composite.
Q3. In an M:N relationship between STUDENT and COURSE with attribute "Grade" on the relationship, where should Grade be placed in the relational schema?
In M:N relationships, a bridge table ENROLLMENT(StdId, CourseId, Grade) is created. Grade belongs here because it's a property of the relationship itself — one student can get different grades in different courses. Putting it in STUDENT or COURSE would cause redundancy.
Q4. A system crashes after a bank transfer transaction has committed. When the system restarts, the transfer is missing. Which ACID property was violated?
Durability guarantees that committed transactions persist even after system failures. If a committed transaction's changes are lost after a crash, Durability was not maintained. Atomicity would be violated if the transaction only partially executed before crashing.
Q5. A hospital has 5 million patient records. Queries for "patients admitted in 2024" are slow. The best partitioning strategy is:
Horizontal partitioning by year places each year's records in a separate partition. A query for 2024 patients only scans the 2024 partition — partition elimination. Hash on name gives uneven distribution for time-based queries. Vertical partitioning splits columns, not useful here. Denormalization increases redundancy but doesn't partition data.
Q6. Which of the following is TRUE about Foreign Keys?
The Referential Integrity Constraint states: FK value must either match a PK value in the referenced (home) relation, OR be completely NULL. FKs CAN be NULL (optional relationship). FK and PK can have different names but must share the same domain. A table can have multiple FKs.
Q7. In the Three-Level ANSI/SPARC architecture, which level is closest to the user and deals with individual user views?
The External Level (also called View Level) is closest to the user. It contains multiple user views — each user sees only the data relevant to them. The Conceptual Level has the full logical structure. The Internal Level has the physical storage structure (indexes, file organization).
Q8. BCNF is violated when:
BCNF rule: For every non-trivial FD X → Y, X must be a candidate key. If any FD has a determinant (LHS) that is NOT a candidate key, BCNF is violated — even if the relation is in 3NF. Having multiple CKs is fine; it's when a non-CK determines something that BCNF breaks.
